## 一、核心用户场景（6 个）

### 场景 1：快速问答（不创建任务）

**用户画像**：李阿姨，58 岁，退休教师，刚学会用电脑，想查个东西。

**用户输入**：`"维生素 D 每天吃多少合适？"`

**系统判断逻辑**：
- 意图分类：知识问答（information-seeking）
- 预估执行时间 < 10 秒
- 不需要文件操作、不需要持续运行
- 结论：**直接在主窗口回答，不创建任务**

**交互流程**：
1. 用户在主窗口底部输入框输入问题，点击发送（或按回车）
2. 输入框上方立即出现用户消息气泡
3. 助手头像旁出现"正在思考..."的呼吸动画（3 个跳动的点）
4. 1-3 秒后，助手消息以打字机效果逐字出现
5. 回答完毕，输入框重新聚焦，等待下一个问题

**关键 UI 状态**：
- 主窗口对话区：正常对话气泡流
- 任务面板：无变化（如果之前没有任务，面板隐藏）
- 输入框状态：发送中 → 灰化不可编辑；回答完毕 → 恢复可编辑

---

### 场景 2：长期开发任务（建网站）

**用户画像**：小张，28 岁，产品经理，不会写代码，想做个个人作品集网站。

**用户输入**：`"帮我做一个个人作品集网站，要简约风格，能展示我的项目截图和简历"`

**系统判断逻辑**：
- 意图分类：创建/构建（creation task）
- 涉及：文件创建、代码编写、多步骤执行
- 预估执行时间 > 60 秒（通常数分钟）
- 结论：**创建任务**

**交互流程**：
1. 用户发送消息
2. 主窗口助手回复（约 2 秒）：

   > "好的，我来帮你做一个简约风格的作品集网站。这个需要一些时间，我会在专门的工作区里完成。"

3. 回复下方自动出现一张**任务卡片**（内嵌在对话流中）：

   ```
   ┌─────────────────────────────────────────┐
   │  [网页图标]  创建个人作品集网站           │
   │  ● 进行中 · 刚刚开始                     │
   │                                          │
   │  正在规划网站结构...                      │
   │  ▰▰▰▱▱▱▱▱▱▱                             │
   │                                          │
   │         [ 查看详情 ]                      │
   └─────────────────────────────────────────┘
   ```

4. 同时，主窗口**右侧任务侧边栏**（如果之前隐藏）自动滑出，显示任务列表，新任务出现在列表顶部
5. 用户可以：
   - **继续在主窗口聊别的**（任务在后台执行）
   - **点击"查看详情"**→ 任务卡片就地展开为完整的任务面板（在主窗口内以抽屉形式从右侧推入，占据约 60% 宽度），显示详细的执行步骤
   - **在侧边栏点击任务条目** → 同上效果
6. 任务执行过程中，侧边栏的任务条目实时更新状态文字：
   - "正在规划网站结构..."
   - "正在创建 HTML 页面..."
   - "正在添加样式..."
   - "正在生成项目截图展示区..."
7. 任务完成后：
   - 系统通知（macOS 原生通知）："你的作品集网站已经做好了"
   - 主窗口对话区追加一条助手消息：

     > "你的作品集网站做好了！包含首页、项目展示页和简历页。文件放在你的工作区 `~/Documents/YiYiClaw/portfolio-site/` 里。"
     >
     > [打开文件夹]  [预览网站]

   - 任务卡片状态更新为"已完成"，带绿色勾号

**关键 UI 状态**：
- 对话区：任务卡片嵌入消息流
- 任务侧边栏：可见，显示进行中任务
- 任务详情面板（可选展开）：显示执行步骤的时间线

### 场景 3：定时任务（每天新闻摘要）

**用户画像**：王先生，45 岁，企业中层管理者，想每天早上快速了解行业动态。

**用户输入**：`"每天早上 8 点帮我整理一份科技行业的新闻摘要"`

**系统判断逻辑**：
- 意图分类：定时/重复任务（recurring task）
- 关键词命中："每天"、"早上 8 点"
- 结论：**创建定时任务**

**交互流程**：
1. 用户发送消息
2. 助手在主窗口回复：

   > "没问题，我会每天早上 8 点帮你整理科技新闻。你可以随时问我要今天的新闻摘要。"

3. 对话流中出现一张**定时任务卡片**（和普通任务卡片有视觉区分——左侧有一个时钟图标和蓝色边线）：

   ```
   ┌─────────────────────────────────────────┐
   │  [时钟图标]  科技新闻每日摘要            │
   │  ⟳ 每天 08:00 · 下次执行：明天 08:00     │
   │                                          │
   │         [ 查看详情 ]  [ 立即执行一次 ]    │
   └─────────────────────────────────────────┘
   ```

4. 第二天早上 8 点，任务自动执行：
   - 系统通知："今天的科技新闻摘要已准备好"
   - 用户打开 App，主窗口顶部出现一个**未读通知横幅**：
     ```
     ┌──────────────────────────────────────────────────┐
     │  [时钟]  科技新闻每日摘要 已完成 · 今天 08:02    │
     │                              [ 查看 ] [ 忽略 ]   │
     └──────────────────────────────────────────────────┘
     ```
   - 点击"查看"，自动在主窗口对话区追加新闻摘要内容
5. 用户也可以在主窗口直接说"今天的新闻呢？"——助手自动关联到这个定时任务，展示最新一次执行结果

**关键 UI 状态**：
- 定时任务卡片使用时钟图标 + 蓝色主题色，区别于一次性任务
- 侧边栏中定时任务有"⟳"循环标识
- 未读执行结果以通知横幅形式提示

### 场景 4：文件处理（分析长报告）

**用户画像**：刘小姐，32 岁，市场分析师，需要快速理解一份长报告。

**用户输入**：用户拖入一个 PDF 文件到输入框，并输入 `"帮我分析这份报告的核心观点，做一个摘要"`

**系统判断逻辑**：
- 意图分类：文件处理（file processing task）
- 输入包含附件（PDF）
- 预估处理时间：取决于文件大小。100 页 PDF → 预估 1-3 分钟
- 结论：**创建任务**（文件超过 5 页时自动创建；5 页以内直接在主窗口处理）

**交互流程**：
1. 用户拖入 PDF，输入框上方显示附件预览缩略图（文件名 + 页数 + 大小）
2. 用户发送消息
3. 助手回复：

   > "这份报告有 98 页，我来仔细阅读并帮你做摘要。大约需要 2 分钟。"

4. 出现任务卡片，进度展示更加细化：
   ```
   ┌─────────────────────────────────────────┐
   │  [文件图标]  分析《2026市场趋势报告》     │
   │  ● 进行中 · 已读取 34/98 页              │
   │                                          │
   │  ▰▰▰▰▱▱▱▱▱▱  35%                       │
   │                                          │
   │         [ 查看详情 ]                      │
   └─────────────────────────────────────────┘
   ```
5. 用户可以在等待时继续和助手聊别的事情
6. 任务完成后，主窗口追加摘要消息，格式清晰：

   > **报告摘要：《2026 市场趋势报告》**
   >
   > **核心观点**：
   > 1. AI 代理市场预计 2027 年达到 XXX 亿...
   > 2. ...
   >
   > **关键数据**：
   > - ...
   >
   > [查看完整分析]  [导出为文档]

**关键 UI 状态**：
- 文件类任务卡片显示处理进度（页数/百分比），比普通任务更直观
- 附件缩略图在输入时预览
- 结果提供"导出为文档"快捷操作

### 场景 5：多任务并行

**用户画像**：小陈，25 岁，自由职业设计师，同时在处理多件事。

**用户输入序列**：
1. `"帮我做一个极简风格的个人网站"`（正在执行中）
2. 30 秒后：`"顺便帮我把这个 PDF 翻译成英文"`（拖入一个 PDF）
3. 再 10 秒后：`"对了，明天下午 3 点提醒我开会"`

**系统判断逻辑**：
- 输入 1：长期任务 → 创建任务
- 输入 2：文件处理任务 → 创建任务
- 输入 3：单次提醒 → 创建定时任务（但更简单，只是一次性提醒）

**交互流程**：
1. 第一个任务创建，侧边栏出现
2. 第二个任务创建，侧边栏列表现在有 2 个条目：
   ```
   ┌── 任务 ──────────────────────┐
   │                               │
   │  ● 创建个人网站               │
   │    正在编写样式... · 2 分钟前  │
   │                               │
   │  ● 翻译 PDF 文档              │
   │    已翻译 12/45 页 · 刚刚     │
   │                               │
   │  ◷ 明天下午 3 点开会提醒       │
   │    将在明天 15:00 触发          │
   │                               │
   └───────────────────────────────┘
   ```
3. 第三个输入，助手在主窗口直接确认：

   > "好的，明天下午 3 点我会提醒你开会。"

   不需要展开成复杂的任务卡片——简单提醒在对话流中用一个小型内联标签即可：

   ```
   [闹钟] 明天 15:00 · 开会提醒   [取消]
   ```

4. 主窗口对话区保持流畅——三条用户消息、三条助手回复依次排列，任务卡片穿插其中但不喧宾夺主
5. 当两个任务都完成后，侧边栏条目变为绿色勾号，点击可查看结果

**关键 UI 状态**：
- 侧边栏任务列表支持同时显示多个任务，按"进行中 → 等待中 → 已完成"排序
- 简单的一次性提醒不使用完整任务卡片，用内联小标签
- 资源提示：如果同时运行的任务超过 3 个，侧边栏顶部出现提示"正在同时处理 3 个任务，速度可能较慢"

---

### 场景 6：从主窗口追问任务进度

**用户画像**：同小张（场景 2），网站任务已经跑了 5 分钟，想看看进展。

**用户输入**：`"网站做得怎么样了？"`

**系统判断逻辑**：
- 意图分类：查询已有任务状态（task status inquiry）
- 关键词匹配 + 语义理解：将"网站"关联到正在运行的任务"创建个人作品集网站"
- 结论：**不创建新任务，在主窗口直接回答并展示任务状态**

**交互流程**：
1. 用户发送"网站做得怎么样了？"
2. 助手自动识别这是在问"创建个人作品集网站"任务的进度
3. 助手回复：

   > "网站正在做，目前进展如下：
   >
   > - [已完成] 规划网站结构
   > - [已完成] 创建 HTML 页面框架
   > - [进行中] 添加 CSS 样式和动画效果
   > - [待开始] 创建项目展示页
   > - [待开始] 部署预览
   >
   > 大约还需要 3 分钟左右。要不要我调整什么？"

4. 如果用户此时说`"项目展示页用卡片布局"`，助手能理解这是对正在执行的任务追加指令：

   > "好的，我会用卡片布局来做项目展示页。已更新任务要求。"

   系统将这条指令注入到对应任务的上下文中。

5. 如果同时有多个任务在运行，用户说`"那个翻译做完了吗？"` → 助手关联到"翻译 PDF 文档"任务

**关键 UI 状态**：
- 主窗口对话中，任务进度以清单形式呈现（不是跳转到任务面板）
- 用户追加的指令通过助手确认后注入任务上下文
- 如果关联不明确（用户说的内容无法确定对应哪个任务），助手主动询问："你是在说网站的事还是翻译的事？"

## 二、UI 布局方案

### 2.1 主窗口整体布局（桌面端，宽屏 1440px+）

```
┌──────────────────────────────────────────────────────────────────┐
│  [窗口标题栏 / 可拖拽区域]                        [—][□][×]     │
├────────┬─────────────────────────────────────┬───────────────────┤
│        │                                     │                   │
│  导    │         主 对 话 区 域               │   任 务 面 板     │
│  航    │                                     │   (可折叠)        │
│  栏    │   [对话气泡流]                       │                   │
│        │   [任务卡片（内嵌）]                  │   ┌─────────────┐│
│  ----  │   [对话气泡流]                       │   │ ● 任务 1    ││
│  [聊天]│   [对话气泡流]                       │   │   进行中...  ││
│  [技能]│                                     │   ├─────────────┤│
│  [定时]│                                     │   │ ● 任务 2    ││
│  [设置]│                                     │   │   45%        ││
│        │                                     │   ├─────────────┤│
│        │                                     │   │ ✓ 任务 3    ││
│        │                                     │   │   已完成     ││
│        │                                     │   └─────────────┘│
│        │                                     │                   │
│        ├─────────────────────────────────────┤                   │
│        │  [附件预览]                          │                   │
│        │  [输入框..............................│.] [发送]         │
│        │                                     │                   │
└────────┴─────────────────────────────────────┴───────────────────┘
```

**布局要点**：
- **左侧导航栏**：保留现有导航（图标 + 文字），但简化为：聊天、技能、自动化、设置。"聊天"是默认且最常用的入口
- **中间主对话区**：占据最大空间（约 60-70% 宽度），是用户的主要交互区域
- **右侧任务面板**：默认隐藏。当有任务创建时自动滑出，宽度约 280px。可以手动折叠/展开（折叠按钮在面板左边缘）
- **输入框**：始终固定在底部，横跨对话区和任务面板。支持拖拽附件、@提及、/ 命令

### 2.2 任务面板详细设计

**折叠状态（无任务时）**：完全隐藏，对话区占满宽度。

**折叠状态（有任务时）**：显示为窄条，仅展示任务数量徽标：
```
┌──┐
│3 │  ← 圆形徽标，点击展开
│  │
└──┘
```

**展开状态**：280px 宽度的侧边栏：
```
┌── 任务 ─────────── [折叠] ──┐
│                              │
│  进行中 (2)                  │
│  ┌──────────────────────┐   │
│  │ [🌐] 创建个人网站     │   │
│  │ 正在添加样式...       │   │
│  │ ▰▰▰▰▰▰▱▱▱▱ 62%     │   │
│  │           [暂停][查看]│   │
│  └──────────────────────┘   │
│  ┌──────────────────────┐   │
│  │ [📄] 翻译PDF文档      │   │
│  │ 已翻译 30/45 页       │   │
│  │ ▰▰▰▰▰▰▰▱▱▱ 67%     │   │
│  │           [暂停][查看]│   │
│  └──────────────────────┘   │
│                              │
│  定时任务 (1)                │
│  ┌──────────────────────┐   │
│  │ [⟳] 科技新闻每日摘要  │   │

│  ✓ 整理会议记录 · 前天       │
│                              │
└──────────────────────────────┘
```

### 2.3 任务详情面板（展开查看）

点击任务的"查看"按钮后，任务详情以**右侧抽屉（drawer）** 的形式推入，覆盖任务列表面板，宽度扩展到约 480px：

```
┌── ← 返回列表 ─── 创建个人网站 ─── [暂停] [取消] ──┐
│                                                     │
│  状态：进行中 · 已运行 4 分 12 秒                    │
│  ▰▰▰▰▰▰▱▱▱▱ 62%                                   │
│                                                     │
│  ── 执行时间线 ──────────────────────────────────   │
│                                                     │
│  ✓ 10:32:01  规划网站结构                           │
│    确定使用单页应用，包含首页、项目、简历三个板块       │
│                                                     │
│  ✓ 10:32:15  创建 HTML 框架                         │
│    创建了 index.html，引入了基础样式                   │
│                                                     │
│  ✓ 10:33:02  编写核心样式                            │
│    极简风格，黑白配色，无衬线字体                      │
│                                                     │
│  ● 10:34:10  添加动画效果（进行中）                   │
│    正在为页面切换添加淡入淡出动画...                   │
│                                                     │
│  ○ 待开始    创建项目展示页                           │
│  ○ 待开始    生成预览                                │
│                                                     │
│  ── 追加指令 ──────────────────────────────────     │
│  [输入框: 对这个任务说点什么...]        [发送]        │
│                                                     │
└─────────────────────────────────────────────────────┘
```

**要点**：
- 任务详情面板底部有独立的输入框，用于向任务追加指令（如"项目展示页用卡片布局"）
- 执行时间线用**步骤节点**的形式展示，类似 GitHub Actions 的日志视图
- 每个步骤可展开查看详细日志（默认折叠，只显示摘要文字）

### 2.4 小屏幕适配（窗口宽度 < 900px）

当窗口较窄时：
- 任务面板**不在侧边栏**，改为**底部抽屉（bottom drawer）**，从底部上滑
- 默认收起状态：底部固定一栏任务状态条（高度 40px），显示"2 个任务进行中"
- 点击状态条 → 上滑展开完整的任务列表
- 任务详情在这种模式下全屏覆盖主对话区（可返回）

### 2.5 移动端适配（未来考虑）

- 主对话区全屏
- 任务通过底部 tab 或顶部下拉面板访问
- 任务详情全屏展示，左划返回

## 三、任务状态机

### 3.1 状态定义

```
                    用户取消
            ┌──────────────────────────────────────┐
            │                                      ▼
         ┌──────┐     ┌─────────┐     ┌──────────────────┐
  创建 → │ 准备中│ ──→ │ 执行中  │ ──→ │     已完成       │
         │PENDING│     │ RUNNING │     │   COMPLETED      │
         └──────┘     └─────────┘     └──────────────────┘
                          │  ▲
                   暂停   │  │ 恢复
                          ▼  │
                      ┌─────────┐     ┌──────────────────┐
                      │ 已暂停  │     │     已失败       │
                      │ PAUSED  │     │     FAILED       │
                      └─────────┘     └──────────────────┘
                          │                    ▲
                          │                    │ 执行出错
                          └──── 取消 ────→  ┌──────────┐
                                            │ 已取消   │
                                            │CANCELLED │
                                            └──────────┘
```

### 3.2 各状态下的用户操作

| 状态 | 界面表现 | 用户可执行的操作 |
|------|---------|----------------|
| **PENDING** | 灰色圆点 + "准备中" | [取消] — 任务未开始就放弃 |
| **RUNNING** | 绿色脉冲圆点 + "进行中" + 进度条 | [暂停] — 暂时中止执行；[取消] — 放弃任务；[追加指令] — 通过输入框向任务注入新要求 |
| **PAUSED** | 黄色圆点 + "已暂停" | [继续] — 恢复执行；[取消] — 放弃任务；[追加指令] — 修改要求后再继续 |
| **COMPLETED** | 绿色勾号 + "已完成" | [查看结果] — 展开查看产出；[重新执行] — 用相同参数重跑一次；[归档] — 从列表中移除 |
| **FAILED** | 红色叉号 + "出了点问题" | [重试] — 重新执行；[查看原因] — 展开错误说明（通俗语言）；[归档] |
| **CANCELLED** | 灰色叉号 + "已取消" | [重新开始] — 用原始要求重新创建任务；[归档] |

### 3.3 状态转换触发条件

| 转换 | 触发条件 |
|------|---------|
| PENDING → RUNNING | 系统资源就绪，Agent 开始执行 |
| RUNNING → COMPLETED | Agent 判断所有步骤完成，产出已生成 |
| RUNNING → FAILED | 执行过程中遇到不可恢复的错误（如 API 超时超过 3 次重试、文件权限被拒绝等） |
| RUNNING → PAUSED | 用户点击"暂停"按钮 |
| PAUSED → RUNNING | 用户点击"继续"按钮 |
| 任意非终态 → CANCELLED | 用户点击"取消"按钮并确认 |

### 3.4 定时任务（Recurring Task）的特殊状态

定时任务不是单次状态机，而是一个**周期性容器**：

```
定时任务状态：ACTIVE（激活） / PAUSED（暂停） / DISABLED（停用）

每次执行是一个子状态机：
  PENDING → RUNNING → COMPLETED / FAILED
```

定时任务的操作：
- [暂停调度] — 暂停后续自动执行，不影响当前正在执行的一次
- [恢复调度] — 恢复自动执行
- [立即执行一次] — 手动触发
- [修改频率] — 更改 cron 表达式（用自然语言："改成每周一"）
- [停用] — 完全停止，从活跃列表移除（可在"已归档"中找回）

## 四、主窗口上下文管理

### 4.1 话题切换处理

**核心原则**：用户不需要手动管理话题，系统自动处理。

**检测机制**：
- Agent 在每轮对话中分析当前消息与上文的语义关联度
- 如果关联度低于阈值（话题明显切换），自动在消息之间插入一个**轻量的话题分隔符**

**UI 表现**：
```
  [助手] 维生素D建议每天补充400-800IU...

  ─ ─ ─ ─  话题分隔线（自动插入，无需用户操作）─ ─ ─ ─

  [你]   帮我做一个网站
  [助手] 好的，我来帮你做...
```

- 分隔线非常轻量：一条虚线 + 可选的时间标记
- 分隔线不需要用户操作，纯视觉引导
- 点击分隔线上方区域可以折叠上一个话题（减少滚动距离）

**上下文压缩**：
- 当主对话消息超过一定长度（如 50 轮），系统自动压缩旧话题为摘要
- 压缩后的话题在 UI 中显示为可展开的折叠区域：
  ```
  ┌──────────────────────────────────────┐
  │  [折叠] 3 月 10 日的话题（2 个）     │
  │  · 维生素D补充量问题                  │
  │  · 明天天气查询                       │
  │           [展开查看原始对话]           │
  └──────────────────────────────────────┘
  ```

### 4.2 历史消息展示

**默认视图：时间线**
- 消息按时间倒序排列（最新的在底部）
- 自动插入日期分隔线："今天"、"昨天"、"3月10日 周一"
- 话题分隔线穿插其中

**可选视图：话题分组**（通过主窗口顶部的视图切换按钮）
- 按话题聚合显示，每个话题是一个可展开的卡片
- 卡片标题自动生成（如"维生素补充咨询"、"个人网站开发"）
- 适合用户回溯查找"之前聊过的某个东西"

```
  顶部栏：[时间线视图 ●]  [话题视图 ○]  [搜索 🔍]
```

### 4.3 关联回溯（"之前那个 XXX"）

当用户说出类似以下表述时，系统需要回溯关联：

- "之前那个网站的事"
- "你上次说的那个方案"
- "早上跟你聊的维生素"

**关联机制**：
1. Agent 提取关键词（"网站"、"方案"、"维生素"）
2. 在以下范围内搜索：
   - 当前主窗口的对话历史（含压缩摘要）
   - 所有任务的标题和描述
   - Memory 系统中的长期记忆
3. 找到匹配后，Agent 自动加载相关上下文继续对话

**UI 辅助**：如果匹配到多个可能的关联，助手会明确询问：

> "你说的是之前做网站的事，还是翻译文档的事？"

如果匹配明确，直接继续，不需要额外确认。

### 4.4 搜索功能

主窗口顶部提供搜索框（放大镜图标），支持：
- 全文搜索对话内容
- 按日期范围过滤
- 按类型过滤（对话 / 任务结果 / 定时任务输出）

搜索结果以列表形式展示，点击跳转到对应位置。

## 五、普通用户友好性检查

### 5.1 需要完全隐藏的概念

| 技术概念 | 处理方式 | 用户看到的是 |
|---------|---------|------------|
| Session | 完全隐藏，用户只看到一个连续的对话 | "和助手的对话" |
| Token / 上下文长度 | 完全隐藏，系统自动压缩 | 无 |
| Model（模型名称） | 隐藏在设置深处，默认自动选择 | 无（高级设置中可选"更聪明但更慢" vs "更快但简单"） |
| Tool / 工具调用 | 默认折叠，只显示"正在帮你处理..."  | "正在读取文件..."、"正在搜索网页..." |
| Agent / ReAct 循环 | 完全隐藏 | 助手在"思考"和"做事"，仅此而已 |
| MCP / Protocol | 完全隐藏 | 无（以"技能"形式呈现给用户） |
| Cron 表达式 | 完全隐藏 | "每天早上 8 点"、"每周一" |
| API Key | 设置页展示为"账号连接"，不展示 key 原文 | "已连接"或"未连接" |
| Working Directory | 隐藏 | "你的文件"（工作区页面入口） |
| 并发数 / 资源限制 | 隐藏，系统自动管理 | 如果任务排队："前面还有 1 个任务，稍等一下" |

### 5.2 任务创建确认策略

**原则：低成本操作自动执行，高成本操作先确认。**

| 任务类型 | 是否需要确认 | 理由 |
|---------|------------|------|
| 快速问答 | 不需要 | 不创建任务，直接回答 |
| 文件分析（只读） | 不需要确认 | 只读操作，无副作用 |
| 创建文件/网站 | **轻确认** | 助手先告知"我来帮你做"，用户有 3 秒窗口可以说"等等"来阻止 |
| 定时任务 | **需要确认** | 涉及周期性执行，助手先展示计划："我会每天 8 点帮你整理新闻，确认吗？" 用户回复"好"或点击确认按钮 |
| 发送消息给别人（Bot） | **需要确认** | 涉及对外通信，必须用户确认 |
| 删除文件 | **强确认** | 弹出确认对话框，明确列出将删除的内容 |

**"轻确认"的交互方式**：
- 不是弹框打断，而是助手回复中带一个隐含的确认窗口
- 助手说"我来帮你做一个网站"后立即开始准备，但在真正执行前给用户约 3 秒时间
- 如果用户在这 3 秒内发送"等一下"或"先别做"，系统中止任务创建
- 如果用户没有反应，自动继续

### 5.3 错误状态的通俗表达

| 技术错误 | 用户看到的信息 | 推荐操作 |
|---------|-------------|---------|
| API rate limit exceeded | "我现在有点忙，稍等一会儿再试" | [稍后重试] 按钮，自动 30 秒后重试 |
| API key invalid | "我的服务连接好像断了。去设置里检查一下？" | [前往设置] 按钮 |
| Network timeout | "网络连接有点不稳定，我再试一次" | 自动重试，3 次失败后显示 [检查网络] |
| File not found | "找不到这个文件，可能已经被移走了" | [重新选择文件] |
| Task execution error | "这个任务遇到了一点问题：[一句话描述]。要我换个方式试试吗？" | [重试] [放弃] |
| Out of memory / context | "这个内容太长了，我需要分批处理" | 自动分批，无需用户干预 |
| Permission denied | "我没有权限操作这个文件/文件夹，需要你授权一下" | 引导用户到系统设置授权 |

**错误展示原则**：
1. 永远不展示技术堆栈、错误码、英文错误信息
2. 用第一人称（"我"）而不是第三人称（"系统"）
3. 每个错误都提供一个明确的下一步操作
4. 可恢复的错误自动重试，不打扰用户

### 5.4 首次使用引导

针对完全不懂 AI 的用户，第一次打开 App 时的引导：

1. **欢迎页**（全屏，温暖的插画风格）：

   > "你好！我是你的 AI 助手。你可以跟我聊天、让我帮你做事情。"
   >
   > [开始使用]

2. **功能暗示**（不做教程，在空白对话区用引导提示）：

   ```
   试试问我：
     "今天天气怎么样？"
     "帮我写一封请假邮件"
     "每天早上帮我整理新闻"
   ```

3. **渐进式发现**：
   - 第一次创建任务时，任务面板滑出的同时显示一个小气泡提示："这里会显示我帮你做的事情的进度"
   - 第一次任务完成时，通知上附带提示："点击查看结果"

### 5.5 无障碍和容错设计

- **输入容错**：用户发送空消息 → 不报错，输入框轻微抖动提示
- **重复发送保护**：连续快速点击发送 → 只处理第一次
- **大文件提醒**：拖入超大文件（>100MB）→ "这个文件比较大，处理起来可能需要较长时间，确定要继续吗？"
- **离开保护**：有任务正在执行时关闭窗口 → "还有 2 个任务在运行，关闭窗口后它们会在后台继续。" [最小化到托盘] [全部停止并关闭]

## 六、设计总结

### 核心设计原则

1. **一个入口**：用户永远只面对一个对话框，不需要理解任何 AI 概念
2. **任务自动浮现**：复杂的事情自动变成可追踪的任务，简单的事情就地解决
3. **不打断，不强制**：任务创建不弹框，确认不阻塞，错误不恐吓
4. **渐进式展示**：默认简洁（只看到对话和任务卡片），需要时才展开（任务详情、执行日志）
5. **通俗语言**：整个界面不出现任何技术术语，用"我"代替"系统"

### 与现有架构的映射

| 新概念 | 现有实现 | 渐进式迁移路径 |
|-------|---------|-------------|
| 主窗口对话 | 默认 Session（隐藏 Session 列表） | 保留 Session 机制，前端只展示一个 |
| 一次性任务 | Session + auto-continue | 标记为 task 类型的 Session |
| 定时任务 | CronJob | 保留现有 CronJob，UI 统一到任务面板 |
| 任务面板 | 无 | 新增组件 |
| 话题分隔 | 无 | Agent 自动插入分隔标记到消息流 |
| 上下文压缩 | compact_messages | 增加话题粒度的压缩 |
