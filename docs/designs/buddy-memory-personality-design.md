## Problem Statement

YiYi 的 Buddy 系统已经有了属性驱动行为、主动问候、记忆浏览、冥想日记、成长里程碑、状态表情、互动统计等功能。但用户能感知到的"成长"还不够明显。成长数据在后台运行，却缺少让用户发出"哇"的可见、可感、有情绪价值的体验。

同时，记忆系统分散在 MemMe (DuckDB) + USER.md + PRINCIPLES.md + SQLite 多个存储中，增加了架构复杂度。

## What Makes This Cool

让 AI 伙伴像女儿一样成长。不是预设的"等级"或"经验值"，而是从真实互动记忆中涌现的人格演化。你和她的每一次对话都在塑造她成为谁。记忆 + 性格 + 能力三位一体，缺一个都不是真正的"成长"。

## Constraints

- 本地优先：Tauri + SQLite 统一存储，去掉 DuckDB 依赖
- LLM 调用是唯一的网络依赖（用户自己的 API key）
- 不引入第三方记忆库（如 Mem0），自建更简洁
- Side project 节奏，一次做一个方向做透

## Premises

1. "能做事的数字伙伴" = Agent + Companion 缺一不可
2. 成长必须可见、可感知、有情绪价值，不只是后台数据

## Out of Scope

- "Life OS" 方向（日历/git/浏览历史整合 → 预判性触达）: 后续 Phase
- "She Noticed" 主动消息系统: 后续 Phase，在统一记忆之上构建
- 声音 / TTS 集成
- 多模态感知（屏幕截图、语音输入）
- 精灵外观/粒子效果的动态变化（仅做最小版本的视觉反馈）

## Cross-Model Perspective

独立 Claude subagent 提出了 "Life OS" 方向：YiYi 读取日历、git commits、浏览历史，构建用户生活的私有模型，做到预判性主动触达。方向认可但列为后续阶段。

核心洞察："记忆+性格+能力"描述的不是产品特性，而是一段关系的成长弧。软件能不能有一个类似人类情感纽带的发展轨迹？这才是真正的赌注。

## Approaches Considered

### Approach A: "成长可视化" (Visible Growth)
聚焦 UI 层面让成长可见：时间线、演化动画、记忆回顾展。
- Effort: M | Risk: Low
- Pros: 直接解决成长可见性，不改底层
- Cons: 不增加实质能力，不解决架构问题。是 Approach B 的子集。

### Approach B: "记忆统一 + 自主人格" (Memory Unification + Autonomous Personality)
先统一存储（去 DuckDB），然后建立基于记忆的人格演化引擎。性格从真实互动中涌现，不靠关键词匹配。包含成长可视化。
- Effort: L | Risk: Med
- Pros: 解决架构问题，人格有真实数据支撑，最接近愿景
- Cons: 工作量较大，MemMe 迁移有风险

### Approach C: "She Noticed" 最小惊艳版
只做一件事：让 YiYi 能主动注意到生活状态并发消息。用现有 scheduler + bots。
- Effort: S | Risk: Low
- Pros: 最快 demo
- Cons: 不解决架构和成长问题

## Recommended Approach

**Approach B: 记忆统一 + 自主人格**

分三个阶段：

### Phase 1: 记忆存储统一 (MemMe → SQLite)

**目标：** 去掉 DuckDB 依赖，所有记忆数据统一到 yiyi.db。

**SQLite Schema:**
```sql
CREATE TABLE IF NOT EXISTS memories (
    id TEXT PRIMARY KEY,              -- UUID
    content TEXT NOT NULL,            -- 记忆内容
    category TEXT NOT NULL,           -- fact | preference | experience | decision | note | principle
    importance REAL DEFAULT 0.5,      -- 0.0-1.0, HOT threshold = 0.7
    keywords TEXT DEFAULT '',         -- 逗号分隔的关键词，用于 FTS 搜索
    source TEXT DEFAULT 'extracted',  -- extracted | user | meditation | correction
    session_id TEXT,                  -- 关联的会话 ID
    created_at TEXT NOT NULL,         -- ISO 8601
    updated_at TEXT NOT NULL,         -- ISO 8601
    decay_factor REAL DEFAULT 1.0     -- 遗忘曲线衰减因子
);

CREATE INDEX idx_memories_importance ON memories(importance DESC);
CREATE INDEX idx_memories_category ON memories(category);

-- FTS5 全文搜索（替代向量搜索）
CREATE VIRTUAL TABLE IF NOT EXISTS memories_fts USING fts5(
    content, keywords, content=memories, content_rowid=rowid
);

-- FTS5 同步触发器（保证 FTS 索引与主表一致）
CREATE TRIGGER memories_fts_insert AFTER INSERT ON memories BEGIN
    INSERT INTO memories_fts(rowid, content, keywords) VALUES (new.rowid, new.content, new.keywords);
END;
CREATE TRIGGER memories_fts_delete AFTER DELETE ON memories BEGIN
    INSERT INTO memories_fts(memories_fts, rowid, content, keywords) VALUES ('delete', old.rowid, old.content, old.keywords);
END;
CREATE TRIGGER memories_fts_update AFTER UPDATE ON memories BEGIN
    INSERT INTO memories_fts(memories_fts, rowid, content, keywords) VALUES ('delete', old.rowid, old.content, old.keywords);
    INSERT INTO memories_fts(rowid, content, keywords) VALUES (new.rowid, new.content, new.keywords);
END;
```

**HOT-tier 机制说明：**
HOT-tier = importance >= 0.7 的记忆（最多 15 条）。在构建 system prompt 时，通过 `SELECT * FROM memories WHERE importance >= 0.7 ORDER BY importance DESC LIMIT 15` 查询，按 category 优先级（principle > decision > preference > fact > note > experience）注入到 prompt context 中。总 token 预算上限 2000 tokens。

**搜索策略：** 使用 SQLite FTS5 全文搜索替代 DuckDB 向量搜索。对于记忆检索场景（用户问"我之前说过什么"、meditation 时回顾），关键词匹配已经足够。不引入 sqlite-vec，保持简洁。**注意：** FTS5 基于关键词匹配，对语义相似但用词不同的记忆（如"和女儿玩"vs"家庭时光"）召回率会低于向量搜索。如果 dogfooding 中发现召回质量不足，后续可添加 sqlite-vec 扩展。抽象边界设计在 memory 检索层，切换搜索后端不影响上层逻辑。

**迁移策略：**
1. 写一次性迁移脚本：读取 DuckDB 全部记忆 → 写入 SQLite memories 表
2. 数据量预估：个人使用场景下，几百到几千条记忆，迁移秒级完成
3. 迁移后验证：对比记忆条数、HOT-tier 内容是否一致
4. 回退方案：保留 DuckDB 文件 30 天不删除，出问题可重新迁移。回退触发条件：迁移后记忆条数差异 > 5%，或任何 HOT-tier（importance >= 0.7）记录缺失
5. 去掉 MemMe / DuckDB 相关 crate 依赖（Cargo.toml）

**保持不变：**
- meditation pipeline 的 5 阶段逻辑不变，只换底层存储调用
- USER.md / PRINCIPLES.md 文件作为 SQLite 的导出/备份保留（SQLite 是 source of truth，文件由 sync_hot_to_files() 单向导出）
- HOT-tier 注入到 system prompt 的逻辑不变
- 记忆提取（extract_memories_from_conversation）流程不变

### Phase 2: 人格演化引擎

**目标：** Buddy 的性格从真实互动记忆中涌现，而不是靠关键词匹配。

**替换 `analyzeGrowth()`：**
现有的 `buddyGrowthKeywords.ts` 用硬编码关键词→属性映射（如包含"代码"→WIT+1）。替换为在冥想时批量分析（不是每次对话后，避免 API 成本和延迟）。

**人格信号表 (personality_signals)：**
```sql
CREATE TABLE IF NOT EXISTS personality_signals (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    trait TEXT NOT NULL,           -- energy | warmth | mischief | wit | sass
    delta REAL NOT NULL,          -- -1.0 到 +1.0 的变化量
    evidence TEXT NOT NULL,       -- 触发这个变化的具体互动摘要
    memory_id TEXT,               -- 关联的 memory ID (FK)
    created_at TEXT NOT NULL
);
```

**分析触发时机：** 在 meditation pipeline Phase C (Growth Analysis) 中执行（每天一次，通过 scheduler cron job 触发，也可手动在 BuddyPanel 点击触发）。读取上次冥想以来的所有新记忆，LLM 分析这些互动对 Buddy 五个属性的影响。LLM 调用失败时（超时、rate limit、JSON 解析错误），跳过本次性格更新，之前的 signals 保持不变，下次冥想重试。输出格式：
```json
{
  "signals": [
    { "trait": "warmth", "delta": 0.3, "evidence": "用户分享了女儿的趣事，互动氛围温馨" },
    { "trait": "wit", "delta": 0.5, "evidence": "用户要求解决复杂编程问题，完成后获得正面反馈" }
  ]
}
```

**属性聚合公式：**
每个属性的当前值 = base_stat（孵化时确定，0-100）+ clamp(SUM(所有该属性 signals 的 delta), -50, +50)。
其中每条 signal 的 delta 范围是 -1.0 到 +1.0，每次冥想最多产生 3 条 signals per trait。
显示值 = clamp(聚合结果, 0, 100)。

**Buddy 回应风格注入：**
当前 `build_system_prompt()` 在 `engine/react_agent/prompt.rs` 中已经注入 SOUL.md + HOT memories + corrections。新增一个 `PersonalityContext` 段落，用上述公式计算当前属性值后注入到 system prompt：
```
你的性格倾向：温柔度 72/100（较高），聪慧 65/100，活力 58/100，调皮 45/100，犀利 30/100（较低）。
你的回应风格应自然反映这些倾向，不需要刻意表演。高温柔意味着更关心用户感受，高聪慧意味着更乐于深入讨论。
```

**信任系统生效：**
当前 `BuddyConfig.trust_scores` 已计算但未使用。在 `engine/buddy_delegate.rs` 的 `delegate()` 和 `delegate_yes_no()` 中加入阈值检查：
- trust_score >= 0.8：Buddy 可自主决策（hosted mode 下自动执行）
- trust_score 0.5-0.8：Buddy 给建议但等待确认
- trust_score < 0.5：Buddy 不主动决策，仅响应直接提问

### Phase 3: 成长可视化 (MVP)

按优先级排序，Phase 3 的 MVP 只做前两项：

1. **成长时间线 UI（P0）：** 在 BuddyPanel 中新增 Timeline tab，展示 personality_signals 的时间轴。每个节点显示：日期、属性变化、触发原因（evidence）。用已有的 `build_growth_timeline()` 数据 + personality_signals 联合展示。

2. **记忆回顾（P0）："还记得那一天……"：** `buddy_observe` 在每次 agent 回复结束时调用（见 `BuddySprite.tsx` 的 `triggerObserve()`，rate-limited by ENERGY stat）。在其触发逻辑中加入 SQL 查询条件：当某条记忆 created_at > 7 天前、importance >= 0.6 时，有 10% 概率在 Buddy bubble 中主动提及。这就是 Success Criterion 5 的实现。

3. **identity traits 展示（P1）：** 在 BuddyPanel 显示 meditation 推断的 identity traits。数据已有（meditation Phase A），只需 UI。

4. **属性变化的最小视觉反馈（Future）：** 属性达到阈值时精灵 glow 颜色微调。不在本迭代范围内。

## Resolved Questions

1. **搜索策略：** FTS5 全文搜索，不用 sqlite-vec。保持简洁，关键词匹配对记忆检索够用。
2. **迁移策略：** 一次性离线迁移，保留旧文件 30 天作为回退。
3. **人格分析时机：** 冥想时批量分析，不是每次对话后。降低 API 成本，集中处理更有质量。

## Open Questions

1. 性格演化的速度怎么调？初步方案：每次冥想最多产生 3 条 signals per trait（每条 delta 范围 ±1.0），总 delta 上限 clamp 在 ±50（对应 0-100 显示刻度上的变化范围）。需要实际使用调参。

## Success Criteria

1. 用户能在 BuddyPanel 的 Timeline tab 中看到 Buddy 的性格变化轨迹和触发原因
2. Buddy 的回应风格会因为 personality_signals 聚合的属性值而自然不同（prompt 注入验证）
3. 去掉 DuckDB 依赖后，cargo build 成功，应用启动正常，记忆数据完整迁移
4. 记忆搜索（FTS5）、HOT-tier 注入、meditation pipeline 在 SQLite 上功能完整
5. Buddy 在使用 > 7 天后，能主动在 bubble 中提及过去的对话（条件：记忆 > 7 天 + importance >= 0.6 + 10% 概率触发）
